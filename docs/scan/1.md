## Findings Críticos

### [CRÍTICO] Estúdio: Dupla instância de `useAudioGenerator` desconecta geração do ActionBar

- **Arquivo:** `src/pages/StudioPage.tsx:51` + `src/App.tsx:154-167`
- **Confidence:** 95/100
- **Evidência:**
  - `StudioPage.tsx:51` cria instância B independente de `useAudioGenerator()`
  - `App.tsx:154-167` cria instância A conectada ao ActionBar
  - `ActionBar` usa instância A (isGenerating, audioUrl, progress)
  - `ScriptEditor` usa instância B (generateAudio)
- **Mitigações Verificadas:** Nenhuma. Não há Context, store compartilhado, ou mecanismo de sincronização.
- **Impacto:** Quando o usuário clica "Gerar áudio", a instância B gera mas a instância A permanece idle. ActionBar retorna null — sem player, sem progresso, sem download. Funcionalidade core quebrada.
- **Sugestão:** Instanciar `useAudioGenerator` uma única vez em `App.tsx` e compartilhar estado/actions com `StudioPage` via props, Context ou Zustand.

### [MÉDIO] Autenticação: Nenhuma verificação de `emailVerified` no fluxo de login

- **Confidence:** 88/100

### [MÉDIO] Dual Storage: Firestore sem persistência offline (`enableIndexedDBPersistence`)

- **Arquivo:** `src/lib/firebase.ts`
- **Confidence:** 85/100
- **Evidência:** Apenas inicialização, sem `enableIndexedDBPersistence()`. Firestore Web SDK usa cache em memória como fallback.
- **Impacto:** Writes pendentes perdidos em refresh/offline. Reads falham completamente offline.
- **Sugestão:** Chamar `enableIndexedDBPersistence(db)` após inicialização. Tratar `failed-precondition` (múltiplas abas).
- **Nota:** Rebaixado de ALTO — cache em memória sobrevive breves desconexões sem refresh.

### [MÉDIO] Autenticação: Exclusão de conta deleta dados antes de remover Auth

- **Arquivo:** `src/contexts/AuthContext.tsx:156-182`
- **Confidence:** 85/100
- **Evidência:** `deleteAllUserData(currentUser.uid)` executa ANTES de `deleteUser(currentUser)`. Se deleteUser falhar, dados já foram removidos irreversivelmente.
- **Impacto:** Perda permanente de dados se `deleteUser` falhar (ex: `auth/requires-recent-login`).
- **Sugestão:** Executar `deleteUser` ANTES, ou implementar `reauthenticateWithCredential` antes da exclusão.

### [MÉDIO] Dual Storage: `deleteChatSession` só deleta Firestore (não IndexedDB)

- **Arquivo:** `src/lib/db/chats.ts:114-124`
- **Confidence:** 90/100
- **Evidência:** Quando `userId` presente, só faz `deleteDoc()` no Firestore sem tocar IndexedDB. `getChatSessions` faz merge de ambos — chat "fantasma" reaparece ao recarregar.
- **Impacto:** Chats que caíram em fallback IndexedDB (>900KB) se tornam indeletíveis pela UI.
- **Sugestão:** Quando autenticado, deletar de ambos (Firestore E IndexedDB) via `Promise.all`.

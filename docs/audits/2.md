## Findings

### [P1-1] Throttle de frame inefetivo no ActionBar

- **Arquivo:** `src/components/ActionBar.tsx:88-97`
- **Tipo:** re-render
- **Confianca:** 92%
- **Descricao:** O `useEffect` tem `bridgeFrame` no array de dependencias (muda ~30x/s via RAF do VideoPreview). Cada mudanca destroi e recria o `setInterval(250ms)` antes que ele dispare. O intervalo usa `useVideoRenderBridge.getState()` (nao closure), o que e correto para leitura de estado, mas o throttle permanece inefetivo pois o interval nunca sobrevive os ~33ms entre mudancas de bridgeFrame. Resultado: ~30 criacoes/destruicoes de intervalo por segundo sem beneficio, e `displayFrame` so e atualizado quando `bridgeIsPlaying` muda para `false` (ao pausar).
- **Impacto:** Alto — 30 criacoes de intervalo/s durante playback + progresso visual congelado durante reproducao
- **Sugestao:** Usar ref para armazenar ultimo frame, removendo bridgeFrame das dependencias do useEffect de throttle.

### [P2-1] Streaming do assistente dispara setState por cada chunk

- **Arquivo:** `src/hooks/useAssistant.ts:250-298`
- **Tipo:** re-render
- **Confianca:** 82%
- **Descricao:** No `for await` do streaming Gemini, cada chunk chama `appendChunkText` que executa `setMessages(prev => prev.map(...))`. Embora React 19 faca automatic batching, `for await` gera microtasks separadas, resultando em renders proximos a taxa de refresh do display (~60fps) durante streaming. Componentes filhos com `React.memo` nao sao protegidos porque o array `messages` sempre muda de referencia. *Nota: throttle com RAF adicionaria latencia perceptivel na UX do chat, portanto a solucao ideal e um buffer leve que batcha chunks dentro do mesmo frame.*
- **Impacto:** Medio — re-renders em cascata durante streaming, mas mitigado por React 19 batching
- **Sugestao:** Buffer Pattern leve com `requestAnimationFrame` — acumular chunks entre frames e aplicar setState uma vez por frame de display. Manter latencia de exibicao <16ms.

### [P2-2] SubtitleOverlay getAlignment() cria objeto inline a cada frame

- **Arquivo:** `src/features/video-render/components/SubtitleOverlay.tsx:184`
- **Tipo:** heavy-computation
- **Confianca:** 82%
- **Descricao:** `getAlignment(position, offsetPadding)` retorna novo objeto `{ justifyContent, alignItems, padding }` a cada frame dentro da Composition Remotion. `position` e `offsetPadding` NAO mudam entre frames.
- **Impacto:** Baixo — ~0.01ms/frame, GC gerencia sem impacto perceptivel
- **Sugestao:** Envolver com `useMemo([position, offsetPadding])`.

### [P2-3] animationStore.progress a 60fps causa re-renders no StrokeRenderer e AnimationControls

- **Arquivo:** `src/features/speed-paint/store/animationStore.ts` + `src/features/speed-paint/components/canvas/AnimationPlayer.tsx:47` + `src/features/speed-paint/components/canvas/AnimationControls.tsx:39`
- **Tipo:** re-render
- **Confianca:** 85%
- **Descricao:** `setProgress(newProgress)` chamado dentro de RAF (~60x/s). StrokeRenderer e AnimationControls subscrevem `progress` diretamente. StrokeRenderer tem mitigacao parcial via `lastRenderedIndexRef` (desenho incremental), limitando o custo real de canvas.
- **Impacto:** Medio — reconciliacao React/Konva a cada frame durante playback de speed-paint, mitigada por desenho incremental
- **Sugestao:** Throttle de `setProgress` para ~15-20fps no AnimationPlayer, ou extrair display de progresso para subcomponente memoizado.

---

### [P2-4] imageProcessing.ts executa edge detection + clusterizacao BFS na main thread

- **Arquivo:** `src/features/speed-paint/lib/imageProcessing.ts:41`
- **Tipo:** heavy-computation
- **Confianca:** 80%
- **Descricao:** Edge detection O(w*h) + BFS com vizinhanca 5x5 rodam na main thread dentro de `setTimeout(..., 50)`. Bloqueia a UI por 500-2000ms. Imagem redimensionada para max 1920x1080 (~2M pixels).
- **Impacto:** Medio — one-shot, usuario espera delay. Nao e continuo.
- **Sugestao:** Mover para Web Worker dedicado para imageProcessing (o strokeWorker existente em video-render nao e reutilizavel diretamente).

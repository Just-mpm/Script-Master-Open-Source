## P1 — Problemas que afetam o usuário diretamente

### [P1-1] Race condition no `useVideoExporter` — refs e estado corrompidos por renderização anterior

- **Fonte:** audit-technical-video-speed → validado 92%+
- **Arquivo:** `src/features/video-render/hooks/useVideoExporter.tsx:310-512`
- **Tipo:** race-condition
- **Problema:** Quando o usuário inicia uma nova renderização enquanto uma anterior está na fase de speed paint, o `finally` da renderização anterior executa `abortControllerRef.current = null` e o `catch` executa `setState({ isRendering: false })`, corrompendo o estado da renderização nova. `generateScenesWithSpeedPaint` não aceita AbortSignal, então a renderização anterior continua e interfere com a nova.
- **Impacto:** `isRendering` revertido para `false` durante renderização real — UI mostra estado incorreto (progresso desaparece, botão de cancelar não funciona). Usuário perde controle da operação.
- **Sugestão:** Adicionar `renderIdRef` (contador incremental) e verificar no catch/finally se a execução ainda é a atual:

```typescript
const renderIdRef = useRef(0);

const startRender = useCallback(async (options) => {
  const renderId = ++renderIdRef.current;
  // ...
  try {
    if (renderIdRef.current !== renderId) return;
    await renderMediaOnWeb({ /* ... */ signal: abortController.signal });
  } catch (err) {
    if (renderIdRef.current !== renderId) return;
    setState(prev => ({ ...prev, isRendering: false }));
  } finally {
    if (renderIdRef.current === renderId) {
      abortControllerRef.current = null;
      speedPaintPhaseWeightRef.current = 0;
    }
  }
}, []);
```

### [P1-2] Chamadas Firestore sequenciais antes do streaming do assistente (150-600ms de latência extra)

- **Fonte:** audit-performance-assistant-studio → validado 95%
- **Arquivo:** `src/hooks/useAssistant.ts:227-238`
- **Tipo:** heavy-compute
- **Problema:** Dentro de `sendMessage()`, `getMemories(user?.uid)` (L227) e `getUserSettings(user?.uid)` (L235) são chamadas sequencialmente via `await` antes do streaming. Ambas acessam Firestore sem cache (getDocs/getDoc direto). Latência soma (50-200ms cada = 150-600ms total).
- **Impacto:** Delay perceptível entre clique de envio e primeiro token. Skeleton fica visível por mais tempo que o necessário.
- **Sugestão:** Paralelizar com `Promise.all`:

```typescript
const [memories, userSettings] = await Promise.all([
  getMemories(user?.uid),
  getUserSettings(user?.uid),
]);
```

### [P1-3] ProtectedRoute — Sem botão "Reenviar email" na tela de verificação

- **Fonte:** audit-ux-flow-pages → validado 88%+
- **Arquivo:** `src/components/ProtectedRoute.tsx:39-54`
- **Tipo:** missing-recovery
- **Problema:** Quando o email de verificação falha silenciosamente ou vai para spam, o usuário é bloqueado na tela "Verifique seu email" com apenas "Voltar ao login" (que leva ao mesmo bloqueio). Sem botão "Reenviar email".
- **Impacto:** Usuário fica em loop sem recovery path. Mensagem "Enviamos um link" é enganosa se envio original falhou.
- **Sugestão:** Adicionar botão "Reenviar email" que chame `sendEmailVerification()` do Firebase Auth, com estado `isResending` e feedback visual.

## P2 — Problemas menores com impacto mensurável

### [P2-1] SpeedPaintControls — `React.memo` quebrado durante drag do slider

- **Fonte:** audit-performance-video-speed → validado 82%+
- **Arquivo:** `src/features/video-render/components/SpeedPaintControls.tsx:53-81` + `src/features/video-render/components/VideoExportPanel.tsx:149,304-305`
- **Tipo:** re-render
- **Problema:** `VideoExportPanel` mantém `speedPaintMultipliers` como estado local. O setter `setSpeedPaintMultipliers` cria um novo objeto `{ ...multipliers, sketch: value }` a cada tick do slider. O `React.memo` no `SpeedPaintControls` faz shallow compare na prop `multipliers`, que falha por referência diferente. Resultado: full re-render ~60x/s durante drag.
- **Impacto:** Componente relativamente leve (2 Sliders + Collapse), mas re-renders desnecessários a cada frame. Impacto perceptível apenas em dispositivos lentos com drag prolongado.
- **Sugestão:** Passar `sketch` e `reveal` como props primitivas separadas:

```typescript
// VideoExportPanel.tsx
<SpeedPaintControls
  sketch={speedPaintMultipliers.sketch}
  reveal={speedPaintMultipliers.reveal}
  onSketchChange={(v) => setSpeedPaintMultipliers(prev => ({ ...prev, sketch: v }))}
  onRevealChange={(v) => setSpeedPaintMultipliers(prev => ({ ...prev, reveal: v }))}
/>

// SpeedPaintControls.tsx — React.memo shallow compare funciona com primitivos
interface SpeedPaintControlsProps {
  sketch: number;
  reveal: number;
  onSketchChange: (value: number) => void;
  onRevealChange: (value: number) => void;
}
```

### [P2-2] Blob URL de `referencePreview` no ImageStudio não é revogado ao desmontar

- **Fonte:** audit-technical-assistant-studio → validado 83%+
- **Arquivo:** `src/components/ImageStudio.tsx:107-128`
- **Tipo:** memory-leak
- **Problema:** `URL.createObjectURL(file)` (L113) armazenado em `referencePreview`. `clearReference` revoga ao remover manualmente, mas não há `useEffect` de cleanup para desmontagem (navegação para outra página).
- **Impacto:** Vazamento de blob URL (poucos KB — ponteiro para File, não dados embutidos) ao navegar sem remover manualmente.
- **Sugestão:** Adicionar cleanup:

```typescript
useEffect(() => {
  return () => {
    if (referencePreview) URL.revokeObjectURL(referencePreview);
  };
}, [referencePreview]);
```

### [P2-3] Library — Falha ao renomear substitui lista de projetos inteira por alerta de erro

- **Fonte:** audit-ux-flow-pages → validado 90%+
- **Arquivo:** `src/components/Library.tsx:224-237`
- **Tipo:** inconsistent-state
- **Problema:** Em `saveEdit`, o `catch` (L234-237) chama `setError(...)` que é a mesma variável de estado do erro de carregamento (L65). A condição ternária L342 (`error ?`) substitui toda a lista de projetos pelo Alert de erro.
- **Impacto:** Usuário perde a visão de todos os projetos ao falhar um rename.
- **Sugestão:** Criar `renameError` separado e exibir feedback inline no TextField (prop `error` do MUI) ou como Snackbar.

### [P2-4] ProtectedRoute loading — Spinner sem aria-label

- **Fonte:** audit-a11y → validado 84%+
- **Arquivo:** `src/components/ProtectedRoute.tsx:24`
- **WCAG:** 4.1.2 Name, Role, Value (Level A)

### [P2-5] DataMigrationDialog — Promise não tratada no useEffect de verificação

- **Fonte:** audit-technical-assistant-studio → validado 92%+
- **Arquivo:** `src/components/DataMigrationDialog.tsx:30-32`
- **Tipo:** error-handling
- **Problema:** `checkForMigratableData().then(setCheckResult)` sem `.catch()`. Se falhar (IndexedDB em Private Browsing do Safari, quota excedida), a rejeição fica sem tratamento.
- **Impacto:** Dialog preso em loading infinito (circular progress) sem mensagem de erro ou retry.
- **Sugestão:** Adicionar `.catch((err) => { log.error('Falha ao verificar dados migráveis', { error: err }); setCheckResult({ hasData: false, summary: { projects: 0, generations: 0, imageGenerations: 0, memories: 0, chats: 0, settings: false } }); })`.

### [P2-6] useAudioGenerator — Sem feedback quando URL de áudio remota falha ao carregar

- **Fonte:** audit-ux-flow-pages → validado 80%+
- **Arquivo:** `src/hooks/useAudioGenerator.ts:215-218`
- **Tipo:** missing-feedback
- **Problema:** `handleError` apenas faz `log.warn` e remove listeners, sem feedback ao usuário. Duração fica em 0 e player mostra 1 frame.
- **Impacto:** Usuário vê player sem duração aparente e sem aviso de que o áudio falhou.
- **Sugestão:** Setar `audioLoadError` state e exibir Alert no VideoPage quando `audioUrl` existe mas duração permanece 0 após timeout.

### [P3-1] Inspector — FileReader sem handler de erro para upload de referência

- **Fonte:** audit-ux-flow-pages → validado 82%+
- **Arquivo:** `src/components/Inspector.tsx:222-228`
- **Problema:** `reader.onerror` não definido. Se `FileReader.readAsDataURL` falhar (arquivo corrompido), falha silenciosa — `onloadend` dispara com `reader.result === null`.

## Findings

### [P1-1] Estúdio de Imagem: sem botão de cancelar durante geração

- **Arquivo:** `src/hooks/useImageGenerator.ts:67-148`
- **Tipo:** missing-feedback
- **Confiança:** 92%
- **Descrição:** O hook `useImageGenerator` não implementa `cancelRef` nem `AbortController`. A chamada à API roda até completar ou falhar (até 30s com retry). O botão muda para "Gerando imagem..." (desabilitado) e exibe skeleton, mas não há forma de cancelar. Compare com o fluxo de TTS que tem botão "Cancelar".
- **Impacto:** Usuário aguarda obrigatoriamente uma operação demorada sem poder interrompê-la. Se cometeu erro no prompt, precisa esperar a geração terminar.
- **Sugestão:** Adicionar `cancelRef` no hook e botão "Parar" no `ImageStudio.tsx` durante `isGenerating`.

### [P1-2] Botão de download ausente para imagens salvas localmente (IndexedDB)

- **Arquivo:** `src/components/ImageStudio.tsx:560`
- **Tipo:** missing-feedback
- **Confiança:** 95%
- **Descrição:** Na galeria, o botão de download só renderiza quando `img.imageUrl` é truthy (`{img.imageUrl && (...)}` — linha 560). Imagens sem login persistem no IndexedDB apenas com `imageBlob`, sem `imageUrl`. A thumbnail aparece via `blobUrls.get(img.id)` (linha 536), mas o botão de download fica invisível. O usuário vê a imagem mas não consegue baixá-la.
- **Impacto:** Funcionalidade core (salvar e baixar) quebrada para uso local. Usuário não logado gera, salva e vê imagens, mas não tem como baixar.
- **Sugestão:** Usar `img.imageUrl || blobUrls.get(img.id)` como fallback para o download.

### [P1-3] Chips do Empty State do assistente são decorativos — não executam ação

- **Arquivo:** `src/features/assistant/components/AssistantMessages.tsx:324-341`
- **Tipo:** confusing-flow
- **Confiança:** 95%
- **Descrição:** O `EmptyChatState` exibe 4 chips (`Ajustar ritmo`, `Sugerir cena`, `Revisar texto`, `Analisar áudio`) com hover states visuais (borda, cor, glow), mas sem `onClick`. São `<Chip>` MUI sem ação. O componente não recebe callback de sugestão.
- **Impacto:** Usuário chega na tela, vê sugestões clicáveis, clica, nada acontece. Afeta negativamente a primeira impressão e pode fazer o usuário achar que está quebrado.
- **Sugestão:** Adicionar `onSuggestionClick` prop e preencher o input com prompt contextual ao clicar.

### [P2-1] Renomear projeto na Biblioteca falha silenciosamente sem try/catch

- **Arquivo:** `src/components/Library.tsx:206-213`
- **Tipo:** missing-error-state
- **Confiança:** 90%
- **Descrição:** `saveEdit` chama `updateProjectName` e `loadProjects` sem try/catch. Se falhar, a exceção não é capturada. `setEditingId(null)` sempre executa, fechando o campo como se funcionasse. O nome "atualizado" na UI é estado local — ao recarregar, o nome antigo reaparece.
- **Impacto:** Feedback visual falso — usuário renomeia, acha que funcionou, mas a mudança não foi persistida.
- **Sugestão:** Envolver em try/catch com feedback de erro e `return` para não fechar o campo de edição.

### [P2-2] Editor de Roteiro: "Limpar" apaga tudo sem confirmação

- **Arquivo:** `src/components/ScriptEditor.tsx:57,176-188`
- **Tipo:** edge-case-unhandled
- **Confiança:** 85%
- **Descrição:** `handleClearScript` executa `setScript('')` diretamente sem confirmação (linha 57). Para roteiros de até 50.000 caracteres, um clique acidental causa perda irreversível. Não há undo (Ctrl+Z não funciona no estado React). Compare com Library e VideoLibrary que sempre pedem confirmação.
- **Impacto:** Usuário pode perder roteiro longo por clique acidental. Frustração alta.
- **Sugestão:** Adicionar `window.confirm` para scripts com mais de ~500 caracteres.

### [P2-3] Inspector: upload de referência sem validação de tamanho

- **Arquivo:** `src/components/Inspector.tsx:194-208`
- **Tipo:** missing-error-state
- **Confiança:** 85%
- **Descrição:** `handleReferenceImageUpload` lê qualquer arquivo via `FileReader.readAsDataURL` sem verificar tamanho. Uma imagem de 20MB+ é convertida para base64 inline no prompt Gemini, podendo causar lentidão, estourar payload ou falhar com erro críptico.
- **Impacto:** Usuário envia imagem grande, geração TTS falha com erro confuso.
- **Sugestão:** Validar `file.size` antes de ler (limite ~10MB).

### [P2-4] Upload de documento no assistente sem try/catch e sem loading

- **Arquivo:** `src/features/assistant/Assistant.tsx:135-163`
- **Tipo:** missing-error-state
- **Confiança:** 88%
- **Descrição:** `handleDocumentUpload` faz `await file.text()` e `await saveMemory()` sem try/catch. Se o arquivo estiver corrompido (encoding inválido), `file.text()` lança erro não tratado. Além disso, não há indicador visual de loading — nem spinner, nem botão desabilitado.
- **Impacto:** Documento corrompido falha silenciosamente. Para arquivos grandes (~500KB), o usuário não tem feedback durante o processamento.
- **Sugestão:** Envolver em try/catch e adicionar estado `isUploadingDocument`.

### [P2-5] Salvar memória curta no assistente sem feedback de loading ou sucesso

- **Arquivo:** `src/features/assistant/Assistant.tsx:165-175`
- **Tipo:** missing-feedback
- **Confiança:** 86%
- **Descrição:** `handleAddMemory` salva via `saveMemory()` sem indicador de loading, confirmação de sucesso ou tratamento de erro (sem catch). Diferente do botão "Salvar insight" nas mensagens (que mostra "Salvo na memória" por 3s), o input lateral não indica se a operação concluiu.
- **Impacto:** Usuário clica "Salvar" sem confirmação. Em caso de falha de rede, a memória parece salva mas não está.
- **Sugestão:** Adicionar estado de loading no botão e feedback visual de sucesso. Adicionar catch.

### [P3-1] "Esqueceu a senha?" e links de navegação usam Typography imperativa em vez de Link

- **Arquivo:** `src/pages/LoginPage.tsx:446-469`, `src/pages/RegisterPage.tsx:434-447`
- **Tipo:** accessibility-issue
- **Confiança:** 84%
- **Descrição:** Links "Esqueceu a senha?", "Cadastre-se" e "Faça login" são `<Typography>` com `onClick={navigate}` e `role="link"` + `onKeyDown`. Não mostra destino no hover do browser, não funciona com Ctrl+Click (nova aba), e `role="link"` sem `href` confunde leitores de tela.
- **Impacto:** Usuários avançados e de leitores de tela são afetados. Funciona para clique simples.
- **Sugestão:** Substituir por `<Link component={RouterLink} to="/cadastro">` do MUI.

### [P3-2] ProtectedRoute exibe spinner sem texto explicativo

### [P3-8] Anexos: break ao encontrar arquivo grande impede validação dos restantes

- **Arquivo:** `src/features/assistant/Assistant.tsx:252-263`
- **Tipo:** edge-case-unhandled
- **Confiança:** 84%
- **Descrição:** No `handleFileChange`, quando um arquivo excede o limite, `break` interrompe o loop. Se o segundo de 3 arquivos for grande, o terceiro (mesmo válido) nunca é avaliado. `setAttachmentError` sobrescreve mensagens anteriores.
- **Impacto:** Usuário com múltiplos arquivos pode perder arquivos válidos que seriam aceitáveis.
- **Sugestão:** Não usar `break`; continuar validando e coletar todos os erros.

### [P3-9] Batch download não mostra progresso por arquivo — apenas spinner global

- **Tipo:** missing-feedback
- **Confiança:** 84%
- **Descrição:** `handleDownloadSequence` baixa áudio + imagens em sequência com único feedback `downloadingId` (spinner). Sem indicação de quantos arquivos foram baixados ou qual está em andamento. Para projetos com 10+ cenas, pode durar dezenas de segundos sem informação de progresso.

### [P3-10] Histórico do assistente: erro de rede exibe empty state sem retry

- **Arquivo:** `src/features/assistant/Assistant.tsx:96-104`
- **Tipo:** missing-error-state
- **Confiança:** 82%
- **Descrição:** `loadHistory` usa try/finally (sem catch explícito). Se `getChatSessions` falhar, o erro vira promise rejection e o drawer fica vazio — indistinguível de "sem histórico". Mesmo padrão em `loadMemories`.

### [P3-11] Copiar mensagem falha silenciosamente sem feedback visual

- **Arquivo:** `src/features/assistant/components/AssistantMessages.tsx:386-394`
- **Tipo:** missing-feedback
- **Confiança:** 85%
- **Descrição:** `handleCopyMessage` faz `navigator.clipboard.writeText()` com catch vazio (comentário: "falha silenciosa"). Na falha, ícone permanece como ContentCopy sem indicação. Usuário assume que funcionou mas nada foi copiado.
- **Impacto:** Em contextos restritos (iframe, HTTP), clipboard falha sem feedback.
- **Sugestão:** Exibir toast "Falha ao copiar" no catch, ou usar fallback `document.execCommand('copy')`.

### [P3-12] Toast de exportação desaparece sem notificação de conclusão

- **Arquivo:** `src/App.tsx:426-467`
- **Tipo:** missing-feedback
- **Confiança:** 83%
- **Descrição:** Snackbar mostra progresso da exportação quando navega para outra rota, mas quando `isExportingVideo` vira `false`, o Snackbar desaparece sem confirmação de sucesso ou link para download. O usuário viu "95%" e então nada.

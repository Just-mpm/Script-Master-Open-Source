## CRITICAL

### [C1] Imagens de cena NÃO são deletadas durante cleanup de conta (LGPD)

**Fonte:** audit-firebase
**Arquivo:** `src/lib/db/account-cleanup.ts:110-113`
**Confidence:** 95%
**Descrição:** O pipeline de exclusão de conta tenta deletar `generations_images/{userId}/{docSnapshot.id}` como arquivo único, mas as imagens reais estão nomeadas como `generations_images/{userId}/{docSnapshot.id}_scene_{index}.png`. O `deleteObject` do Storage deleta apenas um arquivo por chamada — não deleta prefixos. `deleteStorageObjectSafely` engole silenciosamente o erro `object-not-found`.
**Impacto:** Violação LGPD — imagens geradas por IA a partir de roteiros do usuário persistem no Storage após exclusão da conta. A função individual `deleteGeneration()` em `generations.ts:101-109` usa o path correto com `_scene_{index}.png`, mas o cleanup bulk de conta não.
**Evidência:**
```typescript
// account-cleanup.ts:110-111 — path SEM _scene_{index}.png
deleteStorageObjectSafely(
  `generations_images/${userId}/${docSnapshot.id}`, // ❌ errado
  'Erro ao deletar imagens de cena do storage durante cleanup:',
),

// generations.ts:37 — path real COM _scene_{index}.png
`generations_images/${userId}/${generationId}_scene_${sceneIndex}.png` // ✅ correto
```
**Sugestão:** Antes de deletar, ler o doc para obter a quantidade de cenas e iterar sobre os paths corretos, similar ao padrão já usado em `deleteGeneration()`.

### [W6] Exclusão de conta: aviso de limpeza parcial é perdido no redirect

**Fonte:** audit-ux-flow
**Arquivo:** `src/contexts/AuthContext.tsx:166-176`
**Confidence:** 92%
**Descrição:** Quando a exclusão parcialmente falha, `setAuthError()` exibe aviso, mas `deleteUser()` é chamado em seguida. Quando succeed, `window.location.href = '/login'` dispara full reload que destrói estado React antes do toast renderizar. O usuário é redirecionado sem ver que dados residuais permaneceram — informação crítica para LGPD.
**Evidência:**
```typescript
if (cleanupErrors.length > 0) {
  setAuthError(`Alguns dados não puderam ser removidos...`);
  // State update enfileirado, mas render nunca acontece
}
await deleteUser(currentUser); // success → full reload
window.location.href = '/login'; // Destrói estado React
```
**Impacto:** Violação de transparência LGPD — usuário solicita exclusão, acredita que está completa, mas dados residuais permanecem sem que ele saiba.
**Sugestão:** Antes do redirect, exibir `window.confirm()` com o aviso de dados residuais, garantindo que o usuário veja a informação antes da navegação.

### [S0] `handleFirestoreError` perde cadeia de causa original do erro Firebase

**Fonte:** audit-technical (rebaixado de WARNING)
**Arquivo:** `src/lib/db/shared.ts:80-104`
**Confidence:** 84%
**Descrição:** `handleFirestoreError` (declarado `never`) cria `new Error(JSON.stringify(errInfo))` sem preservar o erro original como `cause`. O `errorString` JSON não bate nas regras do `createErrorMapper`, fazendo com que erros de Firestore sempre mostrem a mensagem genérica de fallback para o usuário.
**Impacto:** Usuário nunca vê mensagem amigável específica para erros de Firestore (quota, permissão, rede). Debugging dificultado pela perda do stack trace. Função usada em 13+ locais do projeto. Não quebra funcionalidade — é problema de DX.
**Sugestão:** Preservar o erro original: `throw new Error(errorString, { cause: error })`. Ou mapear o FirebaseError original antes de serializar.

### [S10] Queries Firestore sem `limit()` — carregamento irrestrito em coleções grandes

### [S11] `runRequest` do IndexedDB resolve antes da transação ser commitada

**Fonte:** audit-firebase
**Arquivo:** `src/lib/db/shared.ts:73-78`
**Confidence:** 85%
**Descrição:** `runRequest` resolve no evento `request.onsuccess`, que dispara antes de `transaction.oncomplete`. Se a transação for abortada entre o sucesso individual e a conclusão (quota, erro de outra request), o caller acredita que a operação succeeded quando não foi persistida.
**Mitigações:** Todas as transações no projeto são single-request, exceto `updateIndexedDbItem` (get + put na mesma transação) que é o caso mais suscetível.
**Impacto:** Perda silenciosa de dados em cenários de quota IndexedDB. Na prática, risco baixo.
**Sugestão:** Resolver na conclusão da transação: `transaction.oncomplete = () => resolve(request.result)`.

### [S12] `uploadBlobAndGetUrl` usa upload não-resumável para todos os tamanhos

**Fonte:** audit-firebase
**Arquivo:** `src/lib/db/shared.ts:263-273`
**Confidence:** 82%
**Descrição:** `uploadBlobAndGetUrl` utiliza `uploadBytes` (one-shot) para todos os uploads, incluindo vídeos de até 200MB. O blob inteiro fica na memória antes de enviar, sem progresso, pausa ou retomada.
**Impacto:** Para vídeos (200MB), pode causar OOM em dispositivos com pouca RAM. Sem progresso, usuário não sabe se upload está travado ou em andamento.
**Sugestão:** Usar `uploadBytesResumable` para arquivos acima de 10MB, expondo `onProgress` callback. Manter `uploadBytes` para arquivos pequenos.

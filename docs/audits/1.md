## P1 — Alto

### [V02] EmptyChatState com chips de sugestão nunca renderiza — código morto

- **Severidade:** P1
- **Categoria:** UX Flow
- **Arquivo:** `src/hooks/useAssistant.ts:56` e `src/features/assistant/components/AssistantMessages.tsx:422`
- **Problema:** `messages` é inicializado com `[WELCOME_MESSAGE]` (1 elemento). A condição `isEmptyChat` (`messages.length === 0 && !isLoading`) é sempre `false`. Os 4 chips clicáveis ("Ajustar ritmo", "Sugerir cena", "Revisar texto", "Analisar áudio") são código morto — nunca exibidos. Usuários novos ficam sem orientação ou atalhos contextuais.
- **Impacto:** Perda completa da funcionalidade de onboarding por chips. Primeira experiência significativamente degradada.
- **Confidence:** 98/100
- **Sugestão:** Alterar condição para `messages.length === 1 && messages[0].id === 'welcome' && !isLoading`, renderizando EmptyChatState junto com a mensagem de boas-vindas.

### [T01] `handleFirestoreError` impede fallback para IndexedDB em todas as operações DB

- **Severidade:** P1
- **Categoria:** Error Handling / Data Loss
- **Arquivo:** `src/lib/db/shared.ts:80-104`
- **Problema:** `handleFirestoreError` é declarado com return type `never` e **sempre** lança (`throw new Error(errorString)`). Isso impede que o IndexedDB fallback seja alcançado em todas as funções DB que o chamam dentro de `catch`. Confirmado contra código real: `saveProject`, `getProjects`, `saveAudioToProject`, `saveImageToProject`, `getProjectDetails`, `saveVideoToProject`, `getProjectVideos`, `deleteProject`, `deleteChatSession` — todas seguem o padrão `catch { handleFirestoreError(...) }` seguido de código IndexedDB que é **dead code**.
- **Exceção importante:** `saveChatSession` e `getChatSessions` em `chats.ts` **não** usam `handleFirestoreError` — usam `log.warn` + fallback explícito para IndexedDB. Implementação correta.
- **Impacto:** Qualquer erro transitório do Firestore (rede instável, timeout, permissão temporária) faz a operação falhar completamente em vez de recuar para IndexedDB local. O usuário perde dados que poderiam ter sido salvos localmente — afeta salvamento de projetos, áudios, imagens e vídeos.
- **Confidence:** 92/100
- **Sugestão:** Criar `logFirestoreError()` que apenas loga sem throw, e usar nos catches de funções que têm caminho IndexedDB.

## P2 — Médio

### [V01] Botão "Parar" do composer do Assistente não interrompe streaming

- **Severidade:** P2 *(reclassificado de P1 — stop funcional existe no bubble da mensagem)*
- **Categoria:** UX Flow
- **Arquivo:** `src/features/assistant/components/AssistantComposer.tsx:138-160` e `src/features/assistant/Assistant.tsx:418-427`
- **Problema:** O botão vermelho "Parar" no composer chama `onSubmit` (que é `handleSubmit`), que retorna imediatamente quando `isLoading === true`. O stop funcional existe no bubble da mensagem (`AssistantMessages.tsx:178`), que pode estar scrolled para cima. O composer é fixo na parte inferior — o botão cria falsa affordance.
- **Impacto:** Frustração moderada — usuário clica em "Parar" e nada acontece. Precisa scrollar até o bubble para encontrar o stop real.
- **Confidence:** 96/100
- **Sugestão:** Adicionar prop `onStopGeneration` ao `AssistantComposer` e passá-la em `Assistant.tsx:418`.

### [V03] Exclusão de conta com limpeza parcial não notifica o usuário

- **Severidade:** P2 *(reclassificado de P1 — abordagem best-effort é intencional, código documentado)*
- **Categoria:** UX Flow / Data Loss
- **Arquivo:** `src/lib/db/account-cleanup.ts:88-90` e `src/contexts/AuthContext.tsx:165-172`
- **Problema:** `deleteAllUserData` usa abordagem "best-effort": erros parciais são coletados em `errors[]` e apenas logados. O `deleteUser` do Auth é executado independentemente. Se a deleção do Firestore falhar parcialmente (rede cai), o usuário é redirecionado sem saber que dados residuais ficaram.
- **Impacto:** Dados pessoais podem permanecer no Firestore sem ciência do titular. Questão regulatória LGPD.
- **Confidence:** 92/100
- **Sugestão:** Retornar `errors[]` de `deleteAllUserData`. Se houver falhas, exibir aviso claro: "Não foi possível remover N categorias de dados. Deseja continuar?"

### [V04] Upload de imagem de referência >10MB é rejeitado silenciosamente

- **Severidade:** P2
- **Categoria:** UX Flow
- **Arquivo:** `src/components/Inspector.tsx:206-209`
- **Problema:** Quando `file.size > MAX_REFERENCE_IMAGE_SIZE` (10MB), o código faz `log.warn` + `event.target.value = ''` + `return`. Nenhum feedback visual ao usuário.
- **Impacto:** Frustração direta. Usuário pode tentar múltiplas vezes sem entender por que a imagem não é aceita.
- **Confidence:** 90/100
- **Sugestão:** Disparar `setError()` ou `setSceneGenerationWarning()` com mensagem clara. Exibir helperText indicando o limite de 10MB.

### [V05] ProtectedRoute não preserva URL de retorno ao redirecionar para login

- **Severidade:** P2
- **Categoria:** UX Flow
- **Arquivo:** `src/components/ProtectedRoute.tsx:32-34` e `src/contexts/AuthContext.tsx:68-71`
- **Problema:** Quando usuário não autenticado acessa `/app/biblioteca`, é redirecionado para `/login` sem preservar a URL original. Após login, `onAuthStateChanged` faz `window.location.href = '/app/estudio'` (hardcoded). O state do React Router não sobrevive ao full reload necessário para COEP.
- **Impacto:** Usuário precisa navegar manualmente de volta à página desejada após login. Frustração moderada especialmente em links diretos/compartilhados.
- **Confidence:** 88/100
- **Sugestão:** Persistir URL de retorno em `sessionStorage` antes do redirect. Ler em `AuthContext` após login e redirecionar para ela.

### [V06] Migração de dados marca como concluída mesmo com falhas parciais

- **Severidade:** P2
- **Categoria:** UX Flow
- **Arquivo:** `src/components/DataMigrationDialog.tsx:42-54`
- **Problema:** `handleMigrate` chama `markMigrationCompleted(userId)` no bloco `try` independentemente de `result.errors > 0`. Se 15 de 20 itens migrarem e 5 falharem, o flag de migração é gravado e a oferta nunca aparece novamente.
- **Impacto:** Dados do usuário ficam inacessíveis sem aviso. A caixa de diálogo mostra "15 migrados, 5 erros" mas não oferece retry.
- **Confidence:** 86/100
- **Sugestão:** Só chamar `markMigrationCompleted` quando `result.errors === 0`. Se houver erros, exibir botão "Tentar novamente" para itens restantes.

### [V07] Truncamento silencioso de documento na Base de Conhecimento

- **Severidade:** P2
- **Categoria:** UX Flow
- **Arquivo:** `src/features/assistant/Assistant.tsx:157-162`
- **Problema:** Documentos com `textContent.length > MAX_MEMORY_DOCUMENT_TEXT` (490.000 chars) são cortados via `slice()` sem feedback ao usuário. A validação de arquivo (500KB) não corresponde ao limite de caracteres.
- **Impacto:** O assistente age com informações incompletas sem ciência do usuário. Conteúdo crítico no final do arquivo é perdido silenciosamente.
- **Confidence:** 90/100
- **Sugestão:** Exibir aviso quando truncamento ocorre: "Documento truncado (X de Y caracteres). Apenas a parte inicial foi salva."

### [V08] Batch download (Biblioteca) sem detalhes de falha parcial

- **Severidade:** P2
- **Categoria:** UX Flow
- **Arquivo:** `src/components/video-library/useBatchDownload.ts:26-43`
- **Problema:** O `handleDownloadSequence` faz download sequencial de áudio + cenas. Se qualquer item falhar, o processo inteiro para com erro genérico "Ocorreu um erro durante o download". O usuário não sabe quais arquivos foram baixados com sucesso.
- **Impacto:** Em projetos com 8-15 cenas, falha parcial obriga reinício do zero sem saber o que já tem.
- **Confidence:** 85/100
- **Sugestão:** Adicionar try/catch por item com tracking de `completed/total`. Exibir qual cena falhou e quantas foram concluídas.

## P3 — Baixo

### [V10] Ausência de `beforeunload` durante geração/exportação/gravação

- **Severidade:** P3
- **Categoria:** UX Flow
- **Arquivo:** Ausente em todo o projeto
- **Problema:** Nenhum handler `beforeunload` existe no codebase. Durante geração de áudio, exportação de vídeo ou gravação de batch no Speed Paint, fechar a aba perde todo progresso sem aviso.
- **Impacto:** Perda de trabalho em andamento, mas frequência de fechamento acidental é baixa.
- **Confidence:** 85/100
- **Sugestão:** Adicionar `beforeunload` condicional quando `isGenerating || isExporting || isRecording || batchMode !== 'idle'`.

| [V12] Formulário de contato usa `window.location.href` (mailto) — perda de dados

- **Severidade:** P3
- **Categoria:** UX Flow
- **Arquivo:** `src/pages/public/ContactPage.tsx:294-299`
- **Problema:** `handleSubmit` faz `window.location.href = buildMailtoBody()`, que causa navegação completa (SPA destruída). Se o cliente de email não estiver configurado, a página descarrega e dados preenchidos são perdidos.

### [T02] Mensagens do chat migram para IndexedDB sem notificação ao usuário

- **Severidade:** P3
- **Categoria:** Data Loss / UX Flow
- **Arquivo:** `src/hooks/useAssistant.ts:104-122` e `src/lib/db/chats.ts:44-75`
- **Problema:** `saveChatSession` faz fallback silencioso para IndexedDB quando `estimatedSize > FIRESTORE_MAX_DOC_SIZE_BYTES` (900KB). Chat crescido além do limite é salvo apenas localmente sem qualquer feedback. Em outro dispositivo, o chat não aparece (IndexedDB é local).
- **Impacto:** Conversas grandes ficam disponíveis apenas no dispositivo onde cresceram. Perda silenciosa de sincronização cross-device.
- **Confidence:** 84/100
- **Sugestão:** Exibir indicador visual quando o chat migra para salvamento local: "Conversa salva apenas neste dispositivo."

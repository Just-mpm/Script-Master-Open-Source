### [CRITICAL] Stale closure em `generateAudio` perde estado de áudio anterior

- **Arquivo:** `src/hooks/useAudioGenerator.ts:279`
- **Confidence:** 95/100
- **Tipo:** stale-closure
- **Descrição:** `generateAudio` é definido com `useCallback(..., [])` (deps vazias, linha 632). Na linha 279, `previousState = { audioUrl, audioBlob, scenes, audioSegments }` captura os valores do primeiro render (`null`, `null`, `[]`, `[]`). `lastSuccessfulStateRef.current` é sobrescrito com dados vazios. Quando uma segunda geração falha ou é cancelada, `restoreLastSuccessfulState()` (linhas 612/624) restaura estado nulo/vazio em vez do áudio gerado com sucesso anteriormente. O comentário na linha 207-209 documenta o padrão de deps vazias mas não menciona que `audioUrl`/`audioBlob`/`scenes`/`audioSegments` não são refs — são state values do closure.
- **Impacto:** Usuário perde áudio gerado com sucesso após falha/cancelamento subsequente. Estado restaurado está vazio, não o último válido.
- **Sugestão:** Capturar `audioUrl`/`audioBlob`/`scenes`/`audioSegments` via refs para que `previousState` reflita os valores atuais.

### [CRITICAL] Storage path de vídeo inconsistente no LGPD cleanup

- **Arquivo:** `src/lib/db/account-cleanup.ts:126`
- **Confidence:** 95/100
- **Tipo:** data-integrity (LGPD)
- **Descrição:** O cleanup de vídeos durante exclusão de conta usa `videos/${userId}/${projectId}/${videoDoc.id}.mp4`, mas o upload real em `videos.ts:42` usa `projects/${userId}/${projectId}/videos/${id}.${extension}`. Três erros simultâneos: (1) prefixo `videos/` deveria ser `projects/`, (2) falta nível `videos/` entre projectId e videoId, (3) extensão hardcodada `.mp4` ignora vídeos `.webm`. Como `deleteStorageObjectSafely` captura `object-not-found` silenciosamente, nenhuma exceção é lançada — a falha é invisível.
- **Impacto:** Vídeos do usuário nunca são deletados do Storage durante exclusão de conta LGPD. Viola retenção de dados.
- **Sugestão:** Alinhar o path de deleção com `videos.ts:42`: `projects/${userId}/${projectId}/videos/${id}.${extension}`. Considerar buscar a extensão real do documento ou usar prefix wildcard.

- **Arquivo:** `src/lib/db/account-cleanup.ts:41` + `account-cleanup.ts:160-181`
- **Confidence:** 92/100

- **Descrição:** O cleanup chama `deleteCollectionGroup('image_generations', userId, ...)` que itera sobre a coleção `image_generations` (imagens standalone do Image Studio). ### [WARNING] Deleções de Storage sem await na limpeza de conta (fire-and-forget)

- **Arquivo:** `src/lib/db/account-cleanup.ts:97-101, 110-114, 125-128, 174-177`
- **Confidence:** 95/100
- **Tipo:** race-condition (fire-and-forget)
- **Descrição:** Em `deleteProjectsAndSubcollections` e `deleteCollectionGroup`, as chamadas a `deleteStorageObjectSafely` não são await — a Promise retornada é descartada. Apenas `deleteDoc` é aguardado no `Promise.all`. ### [WARNING] Race condition em `useVideoExporter.startRender` — duplo início sem cancelamento durante speed paint

- **Arquivo:** `src/features/video-render/hooks/useVideoExporter.tsx:311-391`
- **Confidence:** 88/100
- **Tipo:** race-condition
- **Descrição:** O `AbortController` é criado na linha 390, *após* a fase de geração de speed paint (linhas 334-373). Se o usuário iniciar uma nova renderização ou clicar cancelar durante o speed paint, o `abortControllerRef.current` é `null` — a operação continua sem cancelamento. Isso causa: (1) duas gerações de speed paint rodando simultaneamente sem isolamento, (2) `setState` intercalado de ambas corrompendo progresso, (3) cancelamento ineficaz durante a fase mais demorada da exportação.
- **Impacto:** Corrupção de estado de progresso e operações duplicadas consumindo CPU/memória.
- **Sugestão:** Criar `AbortController` antes da fase de speed paint e passar sinal de abort para `generateScenesWithSpeedPaint`.
